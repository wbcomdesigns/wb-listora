01 · The Why

Why Build a New Directory Plugin?

The existing directory market is mature but built on aging foundations — jQuery, post-meta abuse, server-rendered admin AJAX, no REST contract. We're not adding another feature to that pile. We're building the foundation right.

Build the foundation so well that for the next 5 years you only ADD to it, never REWRITE it. With AI, any feature is 1–2 days of work. What takes years to fix is a bad foundation.
— WB Listora product roadmap

Pillar 1 · Performance

Custom tables, not post-meta

Most directory plugins do WP_Query + meta_query for search. That doesn't scale beyond ~5k listings. We use a denormalized search index table with FULLTEXT + a separate field_index for filters + a geo table with geohash. Two-phase search runs in single-digit ms even at 100k rows.

Pillar 2 · API-First

REST is the product, not an afterthought

Every UI action has a REST endpoint. UI is just one consumer — mobile app, Zapier, custom integrations all get the same contract. 74+ documented endpoints under listora/v1, every write hooks before_/after_, every response filterable.

Pillar 3 · Block-First

Interactivity API, zero jQuery

14 Gutenberg blocks built on the official WordPress Interactivity API. Single shared store, server-state hydration, no client/server drift. Themes override block templates WooCommerce-style. Tokens via --listora-* CSS custom properties — themes just override variables.

★
The strategic bet

Competitors have 5–10 years of legacy code. We have 5 years of clean runway. Every architectural decision — custom tables, REST contract, hook surface, template overrides, CSS tokens — is optimized for "we ADD; we don't rewrite."

02 · Architecture

System Architecture · 4 Layers

Strict separation. Blocks consume REST. REST consumes services. Services touch data. Pro plugs into the same hook surface as third parties — it has zero special access.

04 · Under the Hood

How Things Are Actually Done

Three real walk-throughs with file references. So when QA spots a bug, the team can trace it back to a class — not "somewhere in the plugin." All paths below are relative to the active plugin folder.

1

Anatomy of a Search Request

Free

User types in the search box → block fires REST → engine runs two SQL phases → JSON returns. Every hop is observable.

A
Block view-script dispatches
src/interactivity/store.js
Single namespace listora/directory. Search input is debounced 200ms, then store action calls fetch('/wp-json/listora/v1/search?...') with cookie + X-WP-Nonce. No jQuery, no admin-ajax.
B
REST permission + arg validation
includes/rest/class-search-controller.php
Route registered under listora/v1/search. permission_callback returns true for public reads (or WP_Error 401/403 with proper status code on private endpoints). Args sanitized via rest_sanitize_request_arg.
C
Phase 1 · FULLTEXT match
includes/search/class-search-engine.php:224
Single query against listora_search_index: 
MATCH(s.title, s.content_text, s.meta_text)
  AGAINST(%s IN BOOLEAN MODE) AS relevance_score
Indexed by FULLTEXT idx_search declared in includes/class-activator.php:151. Phase 1 returns candidate IDs only, sorted by relevance.
D
Phase 2 · Filter + Geo + Facets
includes/search/class-geo-query.php · class-facets.php
Candidate IDs are intersected with listora_field_index (type-specific filters) and listora_geo (Haversine + bounding-box via geohash prefix). Facet counts computed in same pass — no N+1.
E
Response prep + filter
class-search-controller.php → wb_listora_rest_prepare_search_result
Each result passes through apply_filters('wb_listora_rest_prepare_search_result', $row, $context) so Pro / 3rd parties can add fields without touching core. Pagination via X-WP-Total + X-WP-TotalPages headers.
🧪
How to verify in QA

Open DevTools → Network tab → type in search → see the GET to /wp-json/listora/v1/search. Response should include results[], facets{}, total. To test indexing: run wp listora reindex and watch SELECT COUNT(*) FROM wp_listora_search_index.

2

Anatomy of a Listing Submission

Free + Pro hooks

The hot path everyone tests. Multi-step form → POST to /submission → before-filter (Pro can abort here) → save → after-action (Pro deducts credits, sends emails).

A
Submission block POSTs
blocks/listing-submission/render.php + view-script
Wizard collects type → fields → media → location → terms → guest user (optional) into one payload. Files uploaded via POST /wp-json/wp/v2/media first, IDs included in submission body. reCAPTCHA / Turnstile token attached.
B
REST permission + capability check
includes/rest/class-submission-controller.php:38
permission_callback verifies nonce + capability. Guest submissions handled by separate route — registers user inline if email + name provided. Rate-limit checked here (per-user + per-IP).
C
before_create_listing — abortable
class-submission-controller.php:344
$check = apply_filters('wb_listora_before_create_listing', true, $title, $request); — if any callback returns WP_Error, submission aborts with that error.

Pro hooks here for: credit hold (Credits::hold()), plan-cap enforcement, duplicate guard, moderator routing.
D
Insert post + meta + index
class-submission-controller.php:362
wp_insert_post() → terms assigned → meta written via Meta_Handler → Search_Indexer::reindex_listing($id) rebuilds the row in listora_search_index and listora_field_index → Geo_Service upserts listora_geo.
E
after_create_listing — fan-out
class-submission-controller.php:432
do_action('wb_listora_after_create_listing', $post_id, $request); — fires for: notification emails (Notifications), credit deduction (Pro Credits::deduct()), audit log entry (Pro), outgoing webhook dispatch (Pro), expiration cron registration.
🧪
How to verify in QA

Submit a listing → check (1) post in wp_posts, (2) row in listora_search_index, (3) row in listora_geo, (4) credit row in listora_credit_log if Pro plan-gated, (5) audit row in listora_audit_log if Pro. Email logs via WP Mail Logging plugin or check sandbox inbox.

3

Anatomy of a Block Render

Free + Pro

How a Gutenberg block goes from editor attribute → server HTML → hydrated frontend interactivity. Same pattern across all 14 blocks.

A
block.json registers
blocks/{name}/block.json
apiVersion: 3 · 20 standard attributes (uniqueId, responsive padding/margin, border, shadow, device visibility) · render: file:./render.php · viewScriptModule: file:./view.js (ES module for Interactivity API).
B
render.php builds server HTML
blocks/listing-grid/render.php
Pulls listings via Listing_Data helper (no DB queries in template). Wrapper class generated by Block_CSS::render($attributes) for per-instance CSS scoping. Hooks fired: 
do_action('wb_listora_before_listing_grid', $attrs);
// loop renders cards via wb_listora_get_template()
do_action('wb_listora_after_listing_grid', $attrs);
(see render.php:162 and :169)
C
Server state hydration
render.php:76 → wp_interactivity_state()
wp_interactivity_state('listora/directory', [...]) hands initial state to the frontend store. Critical rule: never define client defaults for server-provided keys — that creates drift on hydration.
D
Frontend store + actions
src/interactivity/store.js
All actions live in the shared store under namespace listora/directory. View.js files import the store to ensure proper load order. No per-block local state — everything observable.
E
Theme override (optional)
{theme}/wb-listora/blocks/{block}/{template}.php
Themes can override any block template WooCommerce-style. Functions: wb_listora_get_template(), wb_listora_locate_template(). Currently used for: listing-card, listing-detail, user-dashboard, all email templates.
🧪
How to verify in QA

View source on any block — wrapper has class wp-block-listora-{name} listora-{uniqueId} with scoped CSS in <style>. DevTools → Application → check window.wp.interactivity.getStore('listora/directory') shows live state. Hooks observable via add_action('wb_listora_after_listing_grid', fn() => error_log('rendered'));

The Six Patterns Every File Follows

Once a tester (or new dev) sees these six patterns, every file in the plugin makes sense.

1
REST controllers extend WP_REST_Controller
One controller per resource. register_routes() in constructor. permission_callback returns WP_Error, never false.
2
before_/after_ on every write
apply_filters('wb_listora_before_*') can abort (return WP_Error). do_action('wb_listora_after_*') for side effects.
3
Custom tables for queryable state
Post meta only for per-listing config. Anything queried across listings (search, geo, reviews) goes in a custom table.
4
Templates via wb_listora_get_template()
Theme-first lookup → plugin fallback. Templates receive $view_data, do zero DB queries — all data prepared in render.php.
5
CSS via --listora-* tokens
Zero hardcoded hex/px in components. Theme overrides by setting tokens in theme.json. Plugin reset isolates from aggressive themes.
6
Pro = same hooks as 3rd parties
Pro has zero special access. Every Pro feature uses the same public hook surface that any extension developer can use.
